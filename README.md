### 📖 프로젝트 소개

**잠깐 시간 될까 - Walk Holic**은 `Spring Boot`와  `flutter`를 사용해서 현재 위치 기반으로 주변 산책로 정보를 제공하고, 사용자 경로 공유 커뮤니티와 같은 여러 편의 기능을 제공해주는 오픈 API 활용 프로젝트입니다.

### ✍🏻 느낀점

- 이번 오픈소스 프로젝트에서 프론트엔드 개발자의 역할을 수행하며 많은 것을 배웠다. 먼저 첫 협업 프로젝트를 진행하면서 나름대로 시간과 노력을 많이 투자 하였지만 프로젝트 경험이 없었던지라 프로젝트 초반에는 공동 작업 환경 설정, 규약 준수, 효과적인 팀 커뮤니케이션과 같은 다양한 측면에서 어려움을 겪었습니다. 이 과정에서 가장 크게 깨달은 점은, 코딩 능력 못지않게 팀원 간의 원활한 소통과 피드백 수용이 프로젝트 성공의 핵심이라는 것이었다.
  
- 백엔드 와의 HTTP 통신
 이 프로젝트에서 백엔드 팀과의 협업은 주로 HTTP 통신을 통해 이루어졌다 HTTP통신은 다양한 HTTP 메소드들(POST, GET, PATCH, DELETE)을 사용하며, 각각의 요청과 응답 과정이 비슷하면서도 독특한 특성을 가지고 동작을 하는 점이 매우 흥미로웠습니다. 백엔드 팀에서 제공한 명확한 API 명세서 덕분에 최신 API를 원활하게 사용할 수 있었고 각 목록 별로 어떤 업데이트가 이루어졌는지 확인하기 좋았다. 프로젝트 경험이 많이 없었던 나에겐 많은 도움 되었고 API 명세서의 중요성을 많이 느낄 수 있었다. 하지만 프론트엔드에서 응답 데이터를 효과적으로 처리하고 저장하는 방법, 그리고 사용자 경험에 영향을 미치는 통신 속도 최적화 문제는 지속적인 고민거리였다. 통신을 통해 원하는 조건의 응답 데이터를 받아도 제대로 다루지 않고 코드의 전개성을 비 효율적으로 처리하게 된다면 사용자 입장에서 그런 디테일적인 불편함이 크게 와닿을것 이라 생각하였다.이러한 문제를 해결하기 위해 팀원들과의 수많은 논의와 협력을 통해 여러 해결책을 모색하였으며 이 과정에서 팀원들의 다양한 아이디어와 지원이 나에게 도움이 많이 되었다.

- UI 디자인 구축
Flutter와 Dart 언어를 사용하여 UI 디자인을 구축하는 과정이 쉽지 않은 부분이었다. Flutter의 다양한 위젯과 그들의 독특한 속성에 대해 학습하며, 앱의 구조를 이해하는 데 많은 시간을 투자하고 노력하였지만 알면 알수록 더욱 시간을 더 쏟아야 하는 것을 알게 되었고 공부하는 과정에서 성취감을 많이 느꼇지만 프로젝트의 진행 과정에서 주요 기능의 추가, 삭제, 수정이 잦았고, 이러한 변화에 신속하게 대응하는 것은 UI 디자인에 더 이상 많은 시간을 투자하기 어려웠다. 일단 기능의 완성도를 높인 후에 UI를 많이 깎아보고 싶었기 때문이다. 이러한 상황에서도 최상의 사용자 경험을 제공하기 위해 꾸준한 노력했으며, 이 과정에서 Flutter의 유연성과 위젯 시스템의 효율성을 깊이 이해할 수 있었다. 앞으로 UI 디자인과 사용자 경험을 더욱 발전시키기 위해 공부를 쉬지 않으면 안되겠다고 생각하였고 UI가 대체로 틀이 잘 잡혀 있어야 사용자들에게도 좋은 모습을 보일 수 있다는 점도 크게 느낄 수 있었다. 

- 어려웠던 부분
  Flutter를 사용하여 카카오 맵 API를 메인 화면에 통합하는 과정에서 여러 기술적인 어려움을 겪었다. 가장 큰 어려움은flutter_webview와 카카오 맵 API 간의 종속성 충돌이었다. 이 문제를 해결하기 위해 Flutter의 버전을 업그레이드하고, pubspec.yaml 파일에서 선언된 패키지 버전을 다운그레이드하는 시도를 했다. 또한, 호환 가능한 다른 위젯을 사용하여 이러한 문제를 극복하기 위해 노력을 하였다. 하지만 이 과정은 시간이 많이 소요되었고, 새로운 tool을 이용하는 과정에서 해결책을 찾는 것이 쉽지 않았다.
  또한, 서비스 내 푸시 알람 기능 구현에 있어서도 어려움을 겪었다. 안드로이드 기기별 버전 차이로 인해 권한 설정에 문제가 발생하였고 이러한 문제는 플랫폼별 특성을 고려한 접근 방식으로 시도 하였다 
  HTTP 통신에서 비동기 함수의 사용은 NULL 처리와 같은 다양한 예외 사항을 관리해야 하였고 이는 특히 Flutter에서 데이터를 가져오고 처리하는 데 있어 중요한 부분이었기 때문에 예외 사항 처리하는데에도 많은 시간을 할애 하였다.
  IOS와 Android에서의 UI 차이는 크로스 플랫폼 개발을 목적으로 하는 우리에게 큰 문제 였다. 두 군데 모두에서 동일한 사용자 경험을 제공하기 위해 그에 맞춰 UI를 조정해야 했다.
-> 이러한 문제들은 팀원들과의 지속적인 의사소통과 협력을 통해 해결할 수 있었다. 각 문제에 대한 해결책을 찾기 위해 팀원들과 의견을 교환하고, 다양한 아이디어를 시도하는 과정에서 많은 것을 배울 수 있었다. 이 경험은 내가 기술적인 문제를 해결하고 뿐만 아니라 팀 내에서 소통과 협력이 가장 큰 부분을 차지 한다고 느낄 수 있게 하였다.

결론,, 협업 프로젝트에선 개개인의 실력도 중요하지만 함께 주기적인 소통을 통해 문제를 해결하고 서로 피드백을 반영하여 하나의 목적으로 향하여 인내하여 끝까지 가는 것이 가장 중요하다!!
